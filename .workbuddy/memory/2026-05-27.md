## C3.5 trip 不可中途打断（天际线式）

### 用户问题
"如果到了指定时间点，worker 还在路上，没有到达目的地，就原地切换目的地，而不是使用瞬移的方式（需要调研一下城市天际线的处理方式）"

### 调研结论（Cities: Skylines）
原文（维基 + Game Developer 专题）：
- "This simulated person would not swerve from their predetermined path unless the route was changed mid-transit, in which case they would be teleported back to their origin instead of calculating a new path from their current location."
- "Unless something happens that causes a given path to no longer be valid or has been extensively modified, a citizen or vehicle will not change their pre-calculated route."

天际线的取舍：trip 是"承诺"，一旦发起就走到底；路径失效不是重算而是 teleport 回起点（它能藏代理，我们藏不了，所以照搬它的 teleport 不可取）。

### 选定方案：方案 1
只允许 AtHome / Working 这种"静止态"响应 phase 切换。已经在路上（GoingToWork / GoingHome）的代理走完当前 trip。

### 修改
tick.ts 第 99 行：
- 旧：`} else if (phase === Phase.Evening && (state[i] === AgentState.Working || state[i] === AgentState.GoingToWork)) {`
- 新：`} else if (phase === Phase.Evening && state[i] === AgentState.Working) {`

去掉了 `|| GoingToWork`，让"早高峰晚到的代理"先到公司打卡再回家，而不是中途 dispatchTrip 触发 wp0 重置导致瞬移。

文件头注释更新到 C3.5，加上"trip 不可打断"的设计依据。

### 长期约定（追加）
- trip 一旦发起就走到底是天际线官方做法，60s/天压缩时间尺度下"晚回家 ~7s"完全可接受
- 只有"静止态"（AtHome / Working）才能响应 phase 切换，禁止"在途"重派发
- 任何"路径中途变更"都是性能/视觉雷区，迟早要走（重算 / teleport / 拐弯）三选一，spike 阶段全部不做

### 构建结果（C3.5）
- TS 零错误，仅 1 行逻辑变化 + 注释
- dev server http://localhost:5180 已重启

## 迭代 2 收尾补丁 · 回填双车道靠右 + 人行道

### 用户反馈
"原来的两车道双向通行与人行道规则呢？"

### 回顾
- C3.4 已经把 sim/roadLayout.ts（ROAD_WIDTH=4 / LANES_PER_DIRECTION=1 / SIDEWALK_WIDTH=0.7 / DEFAULT_DRIVE_OFFSET / SIDEWALK_OFFSET / nsDriveOffset / ewDriveOffset）做完
- C3.4 旧 pathing.ts 在 5 航点 L 形里 inline 调这些偏移，让 walker / driver 各走各侧
- E2 把 pathing.ts 整个重写为 edge 序列 + lerp(from,to)，**车道偏移逻辑直接漂没**
- 视觉症状：路面/双黄线/人行道还在，但代理全部踩在黄线上

### 回填方案
不要把偏移塞进 TripPlan（同一 OD 不同 kind 不能共缓存），改成"投影时按 kind 算"：

新建 src/sim/laneOffset.ts：
- laneOffset(edge, graph, dirAtoB, isDriver) → { dx, dz }
  - NS 路：北行（+z）→ +x；南行（-z）→ -x；off=DEFAULT_DRIVE_OFFSET 或 SIDEWALK_OFFSET
  - EW 路：东行（+x）→ -z；西行（-x）→ +z（与 roadLayout.ewDriveOffset 同口径）
- nodeLanePoint(edge, graph, dirAtoB, isDriver, whichEnd) → 节点中心 + 偏移点（备用）

tick.ts 三处接入：
1. dispatchTrip 末尾把 WalkIn target 从"node 中心"修正到"node + 第一条 edge 的车道入口偏移"——home → 入口点是顺滑斜线，不会先到中线再侧移
2. advanceCruise lerp 投影 + laneOffset(currEdge, ...)
3. 跨 edge / 走完末段时的位置 snap 也加偏移（最后一段 toN.x + off.dx）
4. WalkIn → Cruise 的 phase 切换时 snap 到"node + 偏移"，与 advanceCruise 投影口径一致

### 实现细节
- laneOffset 通过 (b.x - a.x) × (dirAtoB ? 1 : -1) 推行进方向符号，避免每帧重算 sign
- 同一 OD 的 driver / walker 仍共享 TripPlan（偏移不入缓存）
- 转弯（NS→EW 或反向）时方向符号自然翻转：原本北行靠 +x 的车，转东行后靠 -z 是正确"右手边"
- WalkOut 不需要特殊处理：Cruise 最后一帧 snap 到 to + offset 后，WalkOut 朝 work 直线走自动是斜线

### 构建结果
- TS 零错误
- Worker chunk 19.72 → 20.45 KB（+0.7KB，laneOffset 模块）
- 主 chunk 524.25 → 524.50（基本不变）

### 长期约定（追加）
- 车道/人行道偏移是"渲染层细节"，不进 TripPlan 缓存；按代理 kind 在投影时算
- laneOffset 是 sim 内的工具函数，roadLayout.ts 仍是路宽 / 车道宽的真理；laneOffset 把"行进方向 + 偏移量"组合成一次性 (dx, dz)
- 任何"代理位置 = lerp(from, to)"的地方都必须 + laneOffset，否则视觉走中线

## 迭代 2 收尾补丁 · 交叉口卡住 + 镜头近距离剔除

### 用户问题
1. 车辆会卡在十字路口在原地反复移动
2. 镜头推进到一定距离，行人和车辆会不可见

### Bug #1：交叉口反复横跳
tick.ts advanceCruise 跨 edge 的 while 循环里，**edge / eid 变量没同步更新**：
- 旧代码：循环切到下一段后，`edge` 仍指向旧段。下一次循环 / 投影都用错的几何
- arriveNode 用的是旧 edge.to/from，方向判定基于过期数据 → dirAtoB 可能错配
- 投影用旧 edge 的 from/to，但 tOnEdge 已是新段的 t → 位置在两段之间反复跳
- 表现：车辆在井字交叉口"原地移动反复"

修复：循环里同步 `eid = nextEid; edge = nextEdge;`，让循环结束后 edge 就是当前所在段。
顺带把 t 计算改为基于"距离守恒"：`overshootDist = (t-1) × edge.length`，下一段 `t = overshootDist / nextEdge.length`，物理意义清晰。

### Bug #2：镜头推近后实例消失
InstancedMesh 默认按"模板几何 + 第一个实例位置"算 boundingSphere，frustum culling 会把整组当成一个小物体剔除。镜头推近时 boundingSphere 整体掉出视椎 → 全部隐藏。

修复：`this.mesh.frustumCulled = false`（walker / driver 两个 InstanceLayer）。spike 阶段 1k-2k 实例不剔除无性能压力。MVP 期可改为手动设 boundingSphere 覆盖整个 GRID_SIZE。

### 长期约定（追加）
- 任何"循环迭代游走多个数据"的代码，要在循环里同步更新游走指针变量；遗漏会让循环里所有计算 / 判定都基于过期数据
- InstancedMesh 默认 boundingSphere 不可信；spike 阶段直接 frustumCulled=false，正式版手动覆盖整个活动区域
- "Cruise 跨 edge"的 t 换算用"距离守恒"形式（overshootDist / nextLen），不是用 dt 速度等数学等价但难追的复合公式

## 迭代 2 收尾补丁 · 反复抽搐二修（终点 + 倍速）

### 用户反馈
"该问题还是存在，不只是在交叉路口，在道路的终点也会有这个问题"

### 真正的两个根因（之前没找到）

**根因 A：WalkOut/WalkIn 冲过头**
- WalkOut 朝 work 直线推进 `x += vx*DT`，没有检查"剩余距离 < 这一帧能走的距离"
- 4.5×0.25 = 1.125 tile / 帧（车），如果 work 距离 exit node < 1.125 tile 但 > ARRIVE_EPS(0.2)
  → 冲过 target 0.X tile → 下一帧反向走 1.125 tile → 又冲过 → 反复在 target 两侧反弹
- WalkIn 同理（home 紧贴路边时）
- 表现：代理在道路终点 / 进 work 那个点反复抽搐

修：WalkIn / WalkOut 都加 `if (stepDist >= d) snap`，避免冲过头；到达后 x/z 也 snap 到 target。

**根因 B：倍速下 alpha 分母错 + alpha 外推**
- renderTick 用固定 TICK_MS=250ms 做 lerp 分母
- 4x 倍速下 worker 实际每 62.5ms 发 snap → alpha 在 62.5ms 内就到 1，剩余 187.5ms 停在 curr → "走 1/4 时间停 3/4"
- 还有一处 `raw > 1.2 ? 1 : raw`：alpha 允许到 1.2 → 代理冲过 curr 一段，下一份 snap 来时 prev 替换为 curr，视觉反向跳一段
- 叠加 lane offset 在交叉口 NS↔EW 切换时方向符号翻转产生的几何跳变 → 看起来"在交叉口反复抽搐"

修：
- `agents.ts` 加 `snapIntervalMs` 字段，每收到 snap 算"上一份到本份的实际间隔"作 lerp 分母（自动适配 1x/2x/4x）
- alpha 严格 clamp 到 [0, 1]，不再外推（牺牲"快照丢失时的平滑"换"无反向位移"）

### 之前误判的修复（仍保留，是真 bug）
- advanceCruise 循环内同步 `eid = nextEid; edge = nextEdge;`：这是另一个真 bug，但不是用户描述的"反复抽搐"主因
- InstancedMesh frustumCulled = false：与抽搐无关，是另一个独立问题（镜头推近实例消失）

### 长期约定（追加）
- 任何"匀速直线奔目标"的运动都必须有 stepDist >= d 时 snap 的保护，避免越过目标后反弹
- 渲染端 lerp 分母用"上一份到本份的实际间隔"，不用固定 TICK_MS——倍速 / 帧调度抖动都会让两者偏离
- alpha 严格 clamp [0, 1]，不外推；外推带来的"平滑感"在抖动场景下会变成"反向跳"
- "看起来反复"的视觉问题，先怀疑渲染插值的分母 / clamp，再怀疑 sim 层状态震荡——前者更隐蔽
